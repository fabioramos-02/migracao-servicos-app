Resumo do gateway

Gateway MS Digital.

Esforço de migração por tipo

Cada endpoint é classificado pelo esforço de reconstrução: consulta (leitura, o contrato basta), regra (carrega regra de negócio — precisa da ficha) e auth (autenticação/infra técnica). Separa o que é fácil do que é difícil migrar.

O que o Swagger entrega (e o que não entrega)

Reconstruir só pelo Swagger replica a casca e perde a lógica. A migração tem três camadas — o contrato é só uma delas.

Contrato

Paths, métodos, parâmetros, schemas de request/response, auth. O Swagger entrega 100%. É o que o inventário detalha.

Regra de negócio

Validações, orquestração, side-effects, integrações downstream. Não está no Swagger — vem da doc dos módulos do MS Digital, código-fonte e transferência.

Dado persistido

Schema real do banco, campos internos. Parcial — o Swagger só mostra o DTO de borda. Migração de dados é trilha separada.

Estratégia recomendada: strangler fig (migração endpoint a endpoint, gateway roteia old/new) com contrato preservado — transparente para os ~465 mil usuários. Casa com os lotes de migração do Plano PGD-MS.

Priorização por serviço

Ranking dos serviços por volume de endpoints. A barra é relativa ao serviço líder (o 1º enche a trilha); o rótulo traz o nº de APIs e o % do total. A cor indica a faixa de impacto. Na migração strangler fig, a ordem dos lotes deve cruzar este volume com os acessos do BI do APP — serviços mais usados primeiro.

Pendente: integrar os acessos do BI bench-perfil para ranquear por uso real (não só por nº de endpoints). 1º e 2º serviços do lote 1 devem ser arquétipos (cobrir auth/transação), não os mais fáceis.
