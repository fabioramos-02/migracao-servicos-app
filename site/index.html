Levantamento do contrato (Swagger), priorização por uso e estratégia de migração — material de apoio para a reunião de alinhamento.

Resumo do gateway

Gateway MS Digital.

O que o Swagger entrega (e o que não entrega)

Reconstruir só pelo Swagger replica a casca e perde a lógica. A migração tem três camadas — o contrato é só uma delas.

Contrato

Paths, métodos, parâmetros, schemas de request/response, auth. O Swagger entrega 100%. É o que o inventário abaixo detalha.

Regra de negócio

Validações, orquestração, side-effects, integrações downstream. Não está no Swagger — vem da doc dos módulos EDS, código-fonte e transferência.

Dado persistido

Schema real do banco, campos internos. Parcial — o Swagger só mostra o DTO de borda. Migração de dados é trilha separada.

Estratégia recomendada: strangler fig (migração endpoint a endpoint, gateway roteia old/new) com contrato preservado — migração transparente para os ~465 mil usuários. Casa com os lotes de migração do Plano PGD-MS.

Inventário de endpoints

Todos os endpoints do gateway, agrupados por serviço. Filtre por serviço, método ou busque pelo path. Esta é a base objetiva de "qual dado a X-VIA precisa recriar".

Priorização por serviço

Quantidade de endpoints por serviço. Na migração strangler fig, a ordem dos lotes deve cruzar este volume com os acessos do BI do APP — serviços mais usados primeiro.

Pendente: integrar os acessos do BI bench-perfil para ranquear por uso real (não só por nº de endpoints). 1º e 2º serviços do lote 1 devem ser arquétipos (cobrir auth/transação), não os mais fáceis.
